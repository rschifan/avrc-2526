11b — Force layouts

A force layout places nodes by simulating physics: edges act like springs, every node repels every other, a centring force keeps the result on screen, and a custom boundary force keeps it inside the canvas. d3-force ships every piece of this physics in one module.

1. The minimum viable simulation

Three forces are enough for a working layout — plus a hand-written boundary force so nodes don't drift off the visible canvas:

const sim = d3.forceSimulation(nodes)
    .force("link",     d3.forceLink(links).id(d => d.id))
    .force("charge",   d3.forceManyBody())
    .force("center",   d3.forceCenter(innerW / 2, innerH / 2))
    .force("boundary", boundaryForce(innerW, innerH))
    .on("tick", ticked);

function ticked() {
    link.attr("x1", d => d.source.x).attr("y1", d => d.source.y)
        .attr("x2", d => d.target.x).attr("y2", d => d.target.y);
    node.attr("cx", d => d.x).attr("cy", d => d.y);
}

The boundaryForce helper (in _shared/chart.js) is a custom force — a five-line function that clamps every node's x/y to the canvas rectangle on every tick:

function boundaryForce(innerW, innerH, pad = 4) {
    let nodes;
    function force() {
        for (const d of nodes) {
            const r = d.__r ?? pad;
            d.x = Math.max(r, Math.min(innerW - r, d.x));
            d.y = Math.max(r, Math.min(innerH - r, d.y));
        }
    }
    force.initialize = _ => { nodes = _; };
    return force;
}

Any function with an initialize(nodes) method and a callable signature qualifies as a D3.js force — that's the full extension API. Here it just clips coordinates; you could equally implement gravity, an obstacle field, or a custom potential.

boundaryForce hard-clips: nodes that would have drifted past the edge end up squashed against it, which biases the layout near the boundary. The smoother alternative is a pair of gentle attractive forces toward the centre — forceX(cx).strength(0.1) and forceY(cy).strength(0.1) — applied in §2 below.

A different alternative is fit-to-canvas: let the simulation run free and rescale the wrapping <g> after it cools, so the natural bounding box fills the canvas (live demo). Useful as an explicit zoom-to-fit, but node positions are no longer in canvas coordinates so the transform fights any user zoom added later.

2. The four standard forces

In practice you almost always parameterize four forces:

d3.forceSimulation(nodes)
    .force("link",    d3.forceLink(links).id(d => d.id).distance(40))
    .force("charge",  d3.forceManyBody().strength(-150))
    .force("collide", d3.forceCollide().radius(d => r(d) + 2))
    .force("center",  d3.forceCenter(cx, cy));
ForceTunableEffect
forceLink.distance(d) Resting length of the spring on each link.
forceManyBody.strength(s) Negative = repulsion (default -30); more negative spreads the graph out.
forceCollide.radius(r) Hard floor on the gap between node centres.
forceCenter(cx, cy) Anchor point for the layout's centre of mass.
forceX / forceY.strength(s) Per-axis pull toward a target — useful for elongating the layout horizontally.

forceCenter vs forceX + forceY. forceCenter is the obvious choice — "park the graph in the middle of the canvas". It works on this dataset, but it has a known failure mode: on a graph with disconnected components, forceCenter pins the overall centre of mass at (cx, cy) but does nothing to stop individual components from drifting in opposite directions. The modern default, used in the Observable canonical force-directed graph, is forceX(cx).strength(0.1) + forceY(cy).strength(0.1): every node feels a gentle per-axis pull toward the centre, so components stay on-canvas without the centre-of-mass trick. The §3 control panel and the L11 tutorials that follow combine both: forceCenter as a safety net plus forceX/forceY as the actual anchor. For the community-aware version where each node aims at its community's centroid, see 11d §3 — same forces, per-datum target.

3. The interactive control panel

The demo below wires six simulation parameters to live sliders. Each input handler does the same thing — call simulation.stop(), build a fresh forceSimulation with the new parameters, attach a new tick handler:

function rebuild() {
    if (sim) sim.stop();
    sim = d3.forceSimulation(nodes)
        .force("link",    d3.forceLink(links).id(d => d.id).distance(linkDist))
        .force("charge",  d3.forceManyBody().strength(charge))
        .force("collide", d3.forceCollide().radius(4 + collide))
        .force("center",  d3.forceCenter(cx, cy))
        .force("x", d3.forceX(cx).strength(xStr))
        .force("y", d3.forceY(cy).strength(yStr))
        .alpha(0.6)
        .alphaDecay(decay)
        .on("tick", ticked);
}

4. Alpha — how the simulation cools down

A force simulation has an internal alpha in [0, 1] — the temperature. Each tick applies forces scaled by alpha, then alpha decays toward zero. When alpha drops below alphaMin (default 0.001) the simulation stops emitting tick events and fires "end".

sim.alphaDecay(0.0228);   // default — settles in ~300 ticks
sim.alpha(1).restart();   // reheat to 1 and resume
  • Higher alphaDecay (e.g. 0.05) — settles faster, less wiggle, may freeze before a tangled graph untangles.
  • Lower alphaDecay (e.g. 0.005) — runs much longer, lets communities migrate, but feels sluggish on every interaction.
  • Reheat after a structural change (drag, filter, new data) by calling sim.alpha(0.5).restart().

5. Pinning a node

A node with fx, fy set is treated as fixed: it does not move under any force, and other nodes arrange around it. Setting them back to null releases it.

const pinned = nodes.find(d => d.id === "Tyrion-Lannister");
pinned.fx = innerW / 2;
pinned.fy = innerH / 2;

Takeaway. The simulation is the loop, the join is the rendering, the tick handler is the hinge between them. Tune the four standard forces, add a boundary force to keep the graph on canvas, reheat when state changes, and pin nodes by writing fx/fy.
