11a — From NetworkX to D3

In the L7 tutorials we built and analysed networks with NetworkX. The visualizations were produced by matplotlib — static, server-side, one PNG per call. This lecture moves the same network into the browser. The bridge is a JSON file: NetworkX writes it, D3 reads it, the rest is the data join you already know.

1. The node-link JSON format

The de-facto interchange format for networks on the web is a JSON object with two arrays:

{
    "nodes": [
        { "id": "Jon-Snow",      "community": 7, "bc": 0.082 },
        { "id": "Daenerys",      "community": 4, "bc": 0.071 },
        ...
    ],
    "links": [
        { "source": "Jon-Snow", "target": "Tyrion-Lannister", "weight": 5 },
        ...
    ]
}

Each node has an id (any string is fine — names, integers, hashes) and any number of attributes. Each link points at two node ids and may carry attributes of its own (weight, type, year, …).

NetworkX produces this format with a single call:

# in Python
import networkx as nx
import json

data = nx.node_link_data(G)        # → { "nodes": [...], "links": [...] }
with open("got.json", "w") as f:
    json.dump(data, f)

That JSON file is the entire input to every demo in this lecture.

2. The dataset: Game of Thrones co-occurrences

Throughout L11 the running example is a character co-occurrence network from A Song of Ice and Fire: nodes are characters; an edge connects two characters who appear in the same chapter at least once; weight counts how often. The version shipped in ./datasets/got.json includes two precomputed attributes:

  • community — Louvain community label (an integer)
  • bc — betweenness centrality (a number in [0, 1])

The first demo just loads the file and prints a summary — same shape we showed above:

d3.json("./datasets/got.json").then(graph => {
    console.log(graph.nodes.length, "nodes");
    console.log(graph.links.length, "links");
});

3. Drawing the network as raw SVG

A network visualization is just two layers of marks bound to two arrays. With a fixed position for every node, the L9 / L10 data join produces the picture directly — no special layout needed:

const link = svg.selectAll("line").data(graph.links).join("line")
    .attr("x1", d => nodeById.get(d.source).x)
    .attr("y1", d => nodeById.get(d.source).y)
    .attr("x2", d => nodeById.get(d.target).x)
    .attr("y2", d => nodeById.get(d.target).y)
    .attr("stroke", "#999");

const node = svg.selectAll("circle").data(graph.nodes).join("circle")
    .attr("cx", d => d.x)
    .attr("cy", d => d.y)
    .attr("r", 5)
    .attr("fill", "steelblue");

The picture below shows the 20 most central characters with positions assigned by hand on a circle. Edges become <line>; nodes become <circle>. No layout algorithm is involved — the join just maps data to marks at the coordinates we computed.

4. Why a layout is needed

The circular arrangement is readable for 20 nodes because we picked the placement. With 187 nodes the same trick fails: any geometric placement we pick (random, grid, alphabetical-on-a-circle) hides the community structure that is in the data. The picture below drops every character on a random position — same join, same marks, no information.

What we want is for nodes that share many edges to land near each other, and unrelated nodes to drift apart — automatically, from the structure of the graph itself. That's a layout: an algorithm that turns a graph into (x, y) per node. The next tutorial introduces the most common one in D3, d3.forceSimulation.

Takeaway. A network in the browser is two arrays + two joins: links → <line> and nodes → <circle>. Everything we add in 11b–11e — forces, drag, hover, zoom, encoding — sits on top of this skeleton.
