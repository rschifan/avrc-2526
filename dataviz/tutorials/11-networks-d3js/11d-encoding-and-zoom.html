11d — Visual encoding and community overlays

A force layout fixes position; everything else is up to you. Colour carries grouping (community, type, category), size carries importance (centrality, degree, weight), and community hulls draw the groups explicitly. Three short sections, three encoding decisions — that's enough to turn a hairball into a readable network. (Zoom, foreignObject labels, click-to-isolate animation, and the rest of the visual elaborations belong to 11f's progressive build, where they're introduced one at a time on top of these encodings.)

1. Colour by community

The GoT JSON ships a precomputed community integer per node. Map it through an ordinal colour scale and pass the result to fill:

const colour = d3.scaleOrdinal(d3.schemeTableau10);

node.attr("fill", d => colour(d.community));

scaleOrdinal + schemeTableau10 is the default we'll use for any small categorical palette — distinguishable, colour-blind aware, and only ten values to worry about.

2. Size by centrality

Betweenness centrality is highly skewed. Encode it into the area of the circle (not the radius) so the visual signal is proportional to the metric:

const r = d3.scaleSqrt()
    .domain(d3.extent(nodes, d => d.bc))
    .range([3, 16]);

node.attr("r", d => r(d.bc));

Two reasons for scaleSqrt over scaleLinear:

  • SVG's r scales the radius, but humans compare circles by area — scaleSqrt linearises perceived size.
  • Skewed distributions (centrality, populations, weights) benefit from compressing the long tail so the small nodes don't disappear.

3. Cluster forces — make communities emerge from the layout

Colour-by-community shows you the grouping after the fact: the layout doesn't know what a community is, you just paint each circle. A stronger move is to tell the layout about communities so they separate during simulation. The trick is a per-datum forceX / forceY where each node aims at its community's centroid:

// place each community's target on a ring around the canvas centre
const cks = [...new Set(nodes.map(d => d.community))];
const ang = i => (i / cks.length) * 2 * Math.PI;
const rad = 0.30 * Math.min(innerW, innerH);
const tgt = new Map(cks.map((cm, i) => [cm, {
    x: innerW / 2 + rad * Math.cos(ang(i)),
    y: innerH / 2 + rad * Math.sin(ang(i))
}]));

sim.force("xCluster", d3.forceX(d => tgt.get(d.community).x).strength(0.08))
   .force("yCluster", d3.forceY(d => tgt.get(d.community).y).strength(0.08));

Three things to notice:

  • The function argument to forceX is the modern idiom: a single forceX with a per-datum target beats writing N separate forces. Same for forceY.
  • Strength 0.08 is gentle enough that link springs still shape within-community geometry; strong enough that communities visibly separate. Drop to 0 and you get §2's mixed-up layout back; raise to 0.3 and you get rigid blobs.
  • Cluster forces vs forceCenter: a plain forceCenter wants one centre of mass; cluster forces want many. The two compose, but the practical pattern is to drop forceCenter entirely once you have per-community targets — the targets are the anchors. See 11b §2 for the underlying argument about forceCenter on disconnected components.

The result: communities pull apart before you even draw a colour. Use this when grouping is the primary signal you want to communicate; compare to §2 above where the communities are present in the data but blended together by the layout:

For more than ~10 communities, the plugins forceInABox and d3-force-cluster do the same idea with smarter packing. The 11f stage 7 stage shows the standalone version.

4. Convex hulls — drawing the communities

Colour-by-community gets you most of the way; drawing the community boundary finishes the job. d3.polygonHull returns the convex hull of a set of points; feeding the hull through d3.line().curve(d3.curveCatmullRomClosed) turns it into a smooth, closed shape. Recompute on every tick:

const hullPath = d3.line().curve(d3.curveCatmullRomClosed.alpha(0.7));

function drawHulls() {
    const groups = d3.group(nodes, d => d.community);
    const hulls = [...groups].map(([key, members]) => {
        if (members.length < 3) return null;             // hull needs ≥ 3 points
        const pts = members.map(d => [d.x, d.y]);
        return { key, points: d3.polygonHull(pts) };
    }).filter(Boolean);

    hullG.selectAll("path").data(hulls, d => d.key)
        .join("path")
        .attr("d", d => hullPath(d.points))
        .attr("fill", d => colour(d.key))
        .attr("fill-opacity", 0.15)
        .attr("stroke", d => colour(d.key))
        .attr("stroke-width", 1.5)
        .attr("stroke-opacity", 0.6);
}

sim.on("tick", () => { drawHulls(); /* …plus the usual node/link updates */ });

The result lifts the community structure out of the colour and turns it into a first-class visual element. Toggle the hulls on and off to see the effect:

Takeaway. Categorical attribute → scaleOrdinal → colour. Quantitative attribute → scaleSqrt → radius. Group structure has two answers: encode it into the layout via cluster forces (forceX / forceY with per-community targets), or draw it after the layout via d3.polygonHull + curveCatmullRomClosed. Use cluster forces when grouping is the primary signal; use hulls when the layout already separates groups well and you just want to outline them. The 11f progressive build shows each technique standalone.
