11e — Rendering and scale

After 11a–11d you have a working force graph: laid out, interactive, encoded by colour and size. This tutorial covers the three techniques that make such a view hold up in practice — semantic zoom so it stays legible at any zoom level, HTML-quality labels via foreignObject, and the performance levers to reach for as the graph grows. None of these change what the graph is; they change how well it reads and how far it scales.

1. Semantic zoom

Plain d3.zoom on a wrapping <g> scales everything uniformly: zoom in 2× and a 4-pixel circle becomes 8 pixels, a 1-pixel stroke becomes 2, the labels double. That is rarely what you want from a network — you want to get closer to the layout, not magnify the ink. Semantic zoom keeps the geometry zooming while holding the mark sizes constant in screen pixels, and reveals detail (labels) only once you are close enough to read it.

Two ingredients. First, transform the wrapper group so the positions zoom; second, divide every size attribute by the zoom factor k so the ink stays put:

const zoomG = c.plot.append("g");          // every mark lives in here

const zoom = d3.zoom().scaleExtent([0.3, 8])
    .on("zoom", e => {
        const k = e.transform.k, inv = 1 / k;
        zoomG.attr("transform", e.transform);          // positions zoom
        node.attr("r", d => r(d.bc) * inv)             // sizes counter-scale
            .attr("stroke-width", inv);
        link.attr("stroke-width", inv);
        label.attr("dy", d => -(r(d.bc) + 4) * inv)    // offset counter-scales too…
             .style("font-size", `${12 * inv}px`)       // …so does the type
             .style("stroke-width", 3 * inv)
             // reveal labels only past a zoom threshold (plus a curated few)
             .style("display", d => (k >= 1.5 || topLabelled.has(d.id)) ? null : "none");
    });
c.root.call(zoom);
  • The inv = 1 / k trick. The group scale and the per-mark counter-scale cancel, so a circle drawn at r * inv inside a group scaled by k renders at exactly r screen pixels — constant at every zoom level. The same applies to a label's offset: counter-scale its dy by inv too, otherwise the gap above the node grows with k and the label drifts away from its dot as you zoom in.
  • Why not vector-effect: non-scaling-stroke? That handles strokes only — not radii or font sizes. The 1 / k rewrite covers all three uniformly with one number.
  • Zoom-gated labels. k >= 1.5 || topLabelled.has(d.id): zoomed out, only the dozen most-central characters keep their names; zoom past 1.5× and everyone is labelled. This is the cartographer's idea of semantic zoom — more detail emerges as you get closer.

2. Styled labels with foreignObject

SVG <text> is plain text: no background, no padding, no border-radius, no drop shadow. For pill-shaped, coloured, polished labels we borrow HTML's box model by embedding it inside the SVG with <foreignObject> wrapping an xhtml:div:

const fo = zoomG.selectAll("foreignObject").data(topNodes).join("foreignObject")
    .attr("width", 110).attr("height", 22)
    .style("pointer-events", "none");

fo.append("xhtml:div")                         // xhtml: → parsed as HTML, CSS applies
    .style("background", d => colour(d.community))
    .style("color", "white")
    .style("border-radius", "10px")
    .style("padding", "1px 8px")
    .style("box-shadow", "0 1px 2px rgba(0,0,0,0.2)")
    .text(d => d.id.replace(/-/g, " "));
  • The xhtml: prefix matters. Without it the <div> is created in the SVG namespace, where most CSS properties are no-ops. The prefix puts it in the XHTML namespace so the browser lays it out as HTML.
  • Positioning. A foreignObject is placed by its top-left corner, so to centre a label above a node set x = node.x − W/2 and y = node.y − H − r − 2 on every tick.
  • Cost. Each foreignObject is a little HTML subtree the browser must lay out on every move. Lovely for the dozen labels you actually want to spotlight; do not do it for all 187 — there, plain <text> is far cheaper.

3. Performance at scale

The GoT graph (187 nodes) never strains the browser, but the same SVG pipeline will start to judder somewhere in the low thousands of nodes. Before reaching for a different renderer (Canvas / WebGL) or a Web Worker (11f), there are two cheap, SVG-side levers worth knowing — and a third for when labels are present:

// (1) tick budget — only repaint every Nth tick; the simulation still
//     updates positions every tick, so motion stays smooth.
const TICK_BUDGET = 2;
let tickCount = 0;
sim.on("tick", () => {
    if (++tickCount % TICK_BUDGET) return;   // skip the paint this tick
    link.attr("x1", …).attr("y1", …).attr("x2", …).attr("y2", …);
    node.attr("cx", d => d.x).attr("cy", d => d.y);
});

// (2) idle freeze — once alpha < alphaMin the simulation fires "end" and
//     stops emitting ticks on its own; hook it to stop any extra work.
sim.on("end", () => { /* layout is frozen — nothing left to repaint */ });

// (3) label-on-demand — when every label is hidden (zoomed out), skip the
//     per-tick label.attr("x", …) writes entirely.
if (anyLabelVisible) label.attr("x", d => d.x).attr("y", d => d.y);
  • Tick budget roughly halves DOM work for a one-character change. The cost is jitterier motion — drag starts to feel less responsive above ~3.
  • Idle freeze is free: the simulation already stops ticking when it cools, so the per-tick redraw cost drops to zero on its own. The demo below just reports it ("idle (frozen)").
  • Label-on-demand matters once you have many labels — there is nothing to position when they are all hidden, so skip the writes. (The demo below has no labels, so it shows only the first two levers.)

Takeaway. These three techniques are about fit and finish, not new structure. Semantic zoom (group transform + 1/k counter-scale + zoom-gated labels) keeps the view legible at any scale; foreignObject buys HTML typography for a curated label set; the tick budget and idle freeze stretch how far plain SVG carries you. When SVG finally runs out of road, the next move is to take the simulation off the main thread — 11f.
