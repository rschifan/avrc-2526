11f — Progressive build of a network view

L11 has been organised by topic — forces in 11b, interactions in 11c, encoding in 11d. That's a fine reference layout but it hides the fact that a network visualization is built up one capability at a time. This tutorial walks a seven-stage build: stage 1 → stage 2 → stage 3 → stage 4 → stage 5 is the visible progression (empty force → encoded → drag/hover → zoom-aware labels → styled label boxes); stage 6 → stage 7 answers the two production questions that don't appear in the topic-by-topic tutorials — how the layout scales and how community structure can emerge from the layout itself (cluster forces) instead of being painted on after the fact.

Stage 1 — the empty force layout

Adds: nothing. This is the baseline — the data join from 09b on top of d3.forceSimulation from 11b, with default radii and a single colour. 
const sim = d3.forceSimulation(nodes)
    .force("link",   d3.forceLink(links).id(d => d.id))
    .force("charge", d3.forceManyBody())
    .force("center", d3.forceCenter(innerW / 2, innerH / 2));

const link = svg.append("g").selectAll("line").data(links).join("line");
const node = svg.append("g").selectAll("circle").data(nodes).join("circle").attr("r", 4);

Stage 2 — encoded base

Adds: three encodings — node radius from bc (scaleLinear), link stroke width and opacity from weight, node fill from community (scaleOrdinal). 
const r       = d3.scaleLinear().domain(d3.extent(nodes, d => d.bc)).range([5, 15]);
const linkW   = d3.scaleLinear().domain(d3.extent(links, l => l.weight)).range([1, 6]);
const colour  = d3.scaleOrdinal(d3.schemeTableau10);

node.attr("r", d => r(d.bc)).attr("fill", d => colour(d.community));
link.attr("stroke-width", d => linkW(d.weight));

Stage 3 — drag and hover

Adds: d3.drag wired to fx/fy with alphaTarget(0.3).restart(); hover-to-highlight a focal node + its precomputed neighbours. 
node.call(d3.drag()
    .container(() => c.plot.node())
    .on("start", (e, d) => { if (!e.active) sim.alphaTarget(0.3).restart(); d.fx=d.x; d.fy=d.y; })
    .on("drag",  (e, d) => { d.fx = e.x; d.fy = e.y; })
    .on("end",   (e, d) => { if (!e.active) sim.alphaTarget(0); d.fx = null; d.fy = null; }));

node.on("mouseenter", (e, d) => {
    const focus = neighbors.get(d.id);
    node.attr("opacity", n => focus.has(n.id) ? 1 : 0.1);
    link.attr("opacity", l => focus.has(l.source.id) && focus.has(l.target.id) ? 0.9 : 0.05);
});

Stage 4 — zoom-aware labels

Adds: d3.zoom on a wrapping <g> with zoom-invariant sizing — radii, strokes and labels divide by k so they keep their visual size; labels reveal at k ≥ LABEL_THRESHOLD. Wheel to zoom, drag to pan. 
const zoomG = svg.append("g");                  // every mark is inside

d3.zoom().scaleExtent([0.3, 8])
    .on("zoom", e => {
        const k   = e.transform.k;
        const inv = 1 / k;
        zoomG.attr("transform", e.transform);
        node.attr("r", d => r(d.bc) * inv).attr("stroke-width", inv);
        link.attr("stroke-width", inv);
        label.style("display", d => (k >= 1.5 || pinned.has(d.id)) ? null : "none")
             .style("font-size", `${12 * inv}px`);
    });

Stage 5 — styled label boxes

Adds: swaps SVG <text> for <foreignObject> wrapping an xhtml:div — gives the labels a coloured background, rounded corners, padding, real CSS typography. Twelve most-central characters get pill-shaped labels. 
const fo = zoomG.selectAll("foreignObject").data(topNodes).join("foreignObject")
    .attr("width", 110).attr("height", 22)
    .style("pointer-events", "none");

fo.append("xhtml:div")
    .style("background", d => colour(d.community))
    .style("color",       "white")
    .style("border-radius", "10px")
    .style("padding",     "1px 8px")
    .style("font-size",   "11px")
    .style("text-align",  "center")
    .text(d => d.id.replace(/-/g, " "));

Skipped here on purpose: the natural next stages from the legacy sequence are click-to-isolate animation (covered already in 11c §4) and community hulls (11d §3). Re-presenting them would just duplicate material. Instead, the remaining stages add what the topic-by-topic tutorials don't show: how the build scales, and how to make community structure emerge from the layout itself rather than after the fact.

Stage 6 — perf and scale instrumentation

Adds: an FPS counter and three throttling knobs — a tick budget (only redraw every Nth tick), an idle freeze (stop updating the DOM once alpha < alphaMin), and a label-on-demand flag (label positions only update when at least one label is visible). These are the first optimizations to reach for before switching renderer. 
// only repaint every Nth tick
const TICK_BUDGET = 2;
let tickCount = 0;
sim.on("tick", () => {
    if (++tickCount % TICK_BUDGET) return;            // skip
    link.attr(...).attr(...);
    node.attr(...).attr(...);
});

// freeze DOM updates once cool
sim.on("end", () => { /* stop the per-tick redraw */ });

// label work only when visible
if (anyLabelVisible) label.attr("x", d => d.x).attr("y", d => d.y);

Stage 7 — cluster forces (community-aware layout)

Adds: a per-community forceX / forceY that pulls each node toward its community's centroid. Communities separate during layout instead of merely being coloured after the fact — and you no longer need a post-hoc hull to see them. The crucial state-of-the-art tuning that pure forceCenter gets wrong. 
// assign each community a target on a ring
const cks  = [...new Set(nodes.map(d => d.community))];
const ang  = i => (i / cks.length) * 2 * Math.PI;
const cx   = i => innerW / 2 + 0.30 * innerW * Math.cos(ang(i));
const cy   = i => innerH / 2 + 0.30 * innerH * Math.sin(ang(i));
const tgt  = new Map(cks.map((c, i) => [c, { x: cx(i), y: cy(i) }]));

sim.force("xCluster", d3.forceX(d => tgt.get(d.community).x).strength(0.08))
   .force("yCluster", d3.forceY(d => tgt.get(d.community).y).strength(0.08));
Takeaway. Each stage above is one specific technique on top of the previous one. stage 1 → stage 2 → stage 3 → stage 4 → stage 5 covers the visible build of an interactive force-directed network. stage 6 and stage 7 answer the two production questions the topic-by-topic tutorials skipped: how does it scale, and how do communities emerge from the layout itself. From here: 11h moves the simulation off the main thread with a Web Worker for the cases where the single-thread cost is the bottleneck; L12 covers what to do when the node-link picture isn't the right view at all.
