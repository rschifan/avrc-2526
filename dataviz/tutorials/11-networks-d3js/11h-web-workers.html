11h — Web Worker simulation

d3.forceSimulation runs synchronously on the main thread. At scale it competes with rendering, event handlers, and any other JS the page is running, and the UI starts to jitter — drag lags, scroll stutters, transitions skip. A Web Worker runs the simulation on a background thread; the main thread is then free to draw the positions and answer pointer events without contention.

1. What is a Web Worker

A Web Worker is a separate JavaScript context that runs in its own thread. It cannot touch the DOM, but it can run normal JS, import scripts, and communicate with the main thread via postMessage. The classic use case is exactly ours: long-running CPU-bound work that would otherwise block UI updates.

The cost is that data has to be serialised across the boundary. postMessage does a structured clone of whatever you send — slow on large objects, fast on ArrayBuffers (which can be transferred rather than copied, leaving the sender with an empty handle).

2. The protocol

Two files: this page, and _shared/force-worker.js. The protocol is a small set of typed messages:

// main → worker
{ type: "init",       nodes, links, w, h }
{ type: "reheat",     alpha }
{ type: "drag-start", id, x, y }
{ type: "drag-move",  id, x, y }
{ type: "drag-end",   id }

// worker → main
{ type: "links",      endpoints: Uint16Array }    // [s0, t0, s1, t1, …]
{ type: "tick",       positions: Float32Array }   // [x0, y0, x1, y1, …]
{ type: "end" }

Three design choices worth naming:

  • Endpoint table sent once, not per tick. Link source / target indices don't change during a static simulation, so we ship them at init and keep the per-tick payload to positions only.
  • Positions as a typed array. Float32Array instead of an array of objects: half the memory, faster traversal, and — crucially — its backing ArrayBuffer is transferable. The worker hands ownership over; no copy is made.
  • Drag as messages, not as a per-frame call into the worker. The main thread sends a message; the worker writes fx / fy on its copy of the node and the simulation handles the rest.

3. Sending data with Transferables

The second argument to postMessage is a list of objects to transfer instead of clone:

const pos = new Float32Array(nodes.length * 2);
for (let i = 0; i < nodes.length; i++) {
    pos[i * 2]     = nodes[i].x;
    pos[i * 2 + 1] = nodes[i].y;
}
self.postMessage({ type: "tick", positions: pos.buffer }, [pos.buffer]);

Without the second argument, every tick would clone the entire array. At 10k nodes that's 80 KB cloned 60 times per second — measurable. With the transfer, the buffer hops to the main thread in O(1).

4. Reading the positions on the main thread

worker.onmessage = (e) => {
    const m = e.data;
    if (m.type === "links") endpoints = new Uint16Array(m.endpoints);
    if (m.type === "tick")  { positions = new Float32Array(m.positions); draw(); }
    if (m.type === "end")   { /* simulation cooled */ }
};

// One  per edge and one  per node, created once at init.
// On every tick we just update attributes — no DOM churn.
function draw() {
    if (!positions || !endpoints) return;
    lines
        .attr("x1", (_, i) => positions[endpoints[i*2]     * 2])
        .attr("y1", (_, i) => positions[endpoints[i*2]     * 2 + 1])
        .attr("x2", (_, i) => positions[endpoints[i*2 + 1] * 2])
        .attr("y2", (_, i) => positions[endpoints[i*2 + 1] * 2 + 1]);
    circles
        .attr("cx", (_, i) => positions[i * 2])
        .attr("cy", (_, i) => positions[i * 2 + 1]);
}

The main thread keeps two typed arrays around (endpoints and positions) and writes attributes from them — no node objects, no link objects, no lookup table for the rendering loop. Same SVG marks as every other demo in L11; the only difference is where the positions come from.

6. When to introduce a worker

  • Not before you need it. A worker doubles the moving parts: separate file, message protocol, serialisation. For under ~1k nodes the main-thread version is simpler and just as fast.
  • Add it when the simulation noticeably blocks UI: scrolling stutters during warmup, drag feels laggy, layout-heavy pages co-located with the chart judder. The trigger is main-thread responsiveness, not raw simulation speed.
  • The hard parts: shipping updates to the simulation (new nodes, removed nodes, attribute changes) needs careful protocol design. The dynamic-graph version is covered in 12f in the next class.
Takeaway. The simulation doesn't need the DOM and the rendering doesn't need the simulation. A Web Worker splits them onto two threads; Transferable typed arrays make the message channel cheap. The result: the layout converges in the background while the main thread stays responsive to drag, hover, scroll, and any other UI work the page is doing.
