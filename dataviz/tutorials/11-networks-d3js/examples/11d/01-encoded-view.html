11d §1-§2 — Colour by community, size by centrality

Two encoding decisions turn a hairball into a readable network: hue carries group membership, area carries importance. Both are one-liners on top of the layout machinery from 11b. Source tutorial →

What this example shows

A force layout fixes position; everything else is up to the encoding. Two scales do the work: an ordinal scale mapping community to a categorical palette, and a sqrt scale mapping betweenness centrality to circle radius (so the visual signal grows by area, not radius).

  • d3.scaleOrdinal(d3.schemeTableau10) — the default 10-colour categorical palette. Distinguishable, colourblind-aware, and only ten values to worry about. Pass an integer/string and get back a colour string.
  • Why scaleSqrt instead of scaleLinear: humans compare circles by area, not radius. Since area = π · r², mapping bc through scaleSqrt (which raises to the 0.5) and then assigning to r gives a perceived size that grows linearly with the metric. The second reason: centrality distributions are heavy-tailed, and the square root compresses the long tail so small-degree nodes don't disappear.
  • scaleSqrt().domain(d3.extent(nodes, d => d.bc)).range([3, 14]) — minimum 3px so even the least-central node is visible; maximum 14px so the most central one doesn't overwhelm the layout. Both numbers are visual judgements, not derived from the data.
  • Re-tune the collide radius: forceCollide().radius(d => r(d.bc) + 1). With variable node sizes, a constant collide radius would either let large nodes overlap or push small ones too far apart.
  • Native SVG tooltip via <title>: the browser renders it on hover. No custom HTML overlay; useful as a baseline before building a styled tooltip.
