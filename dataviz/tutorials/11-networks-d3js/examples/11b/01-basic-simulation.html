11b §1 — Minimum viable force simulation

Three built-in forces + a hand-written boundary force are enough for a working layout. The simulation iterates; on every tick we copy node positions onto the SVG marks. Source tutorial →

What this example shows

A force layout simulates physics: edges act like springs, every node repels every other, a centring force keeps the result on screen. The simulation runs ~300 ticks, cools down, and emits an "end" event — by then the graph has "relaxed" into a shape that reflects its connectivity.

  • d3.forceSimulation(nodes) is the iterator. You attach forces with .force(name, force); you read positions in a "tick" handler. The tick handler is the hinge between the simulation (which mutates d.x, d.y) and the data join (which renders them).
  • Standard forces:
    • forceLink: springs along edges. .id(d => d.id) tells it how to resolve link endpoints to node objects.
    • forceManyBody: pairwise repulsion (default -30).
    • forceCenter: anchors the centre of mass.
  • The warmup overlay (attachWarmup(svg, sim)) hides the layout while it converges — without it, students see nodes jitter from random positions into the final shape, which is confusing on first viewing.

This is the one place we use fitToCanvas. The simulation above has no boundary — only forceCenter. Without something else, the layout can drift past the canvas edges. fitToCanvas(wrap, sim, nodes, innerW, innerH) attaches to the simulation's "end" event: it computes the bounding box of node positions and applies a single transform to the wrapping <g> so the graph snaps into view. A useful one-off zoom-to-fit, but not the default for L11 — see the tutorial's §1 for the alternative pattern (boundaryForce, a custom D3 force that clamps positions tick-by-tick). Every other L11 demo uses neither: just a gentle forceX(cx).strength(0.1) + forceY(cy).strength(0.1) pair, the modern default, which keeps the layout on canvas without either an after-the-fact transform or a hard clip.
