11f — Web Worker simulation

A worker doesn't make a force layout compute faster — it keeps the main thread free so the page stays responsive while the layout runs. To feel that, watch the spinner: run the layout on the main thread and it stutters; run the same layout in a worker and it keeps turning. Source tutorial →

What this example shows

One stress graph (a synthetic random network — not meant to be read, just to be expensive), laid out by the same simulation two ways. The spinner is driven by requestAnimationFrame, so it only turns smoothly when the main thread is free to paint. The worst frame gap readout measures the longest stall during the run (one-time setup cost excluded).

  • Run on the MAIN thread — each force tick blocks the next frame; the spinner hitches and the worst gap climbs to tens or hundreds of milliseconds.
  • Run in the WORKER — the force step is off-thread; the main thread only paints positions, so the spinner stays smooth and the gap stays near one frame (~16 ms).
  • It's about responsiveness, not speed. Read the spinner and the gap, not the simulation's fps — the worker may even tick a little slower (message overhead). Crank the node count up if the contrast is subtle on a fast machine.
  • What is not offloaded: rendering. Both runs write SVG attributes on the main thread every tick — only the force step moves to the worker. A render-bound view needs a different fix (Canvas / WebGL).
  • Worker prerequisite: serve over HTTP, not file://. Use python -m http.server.
