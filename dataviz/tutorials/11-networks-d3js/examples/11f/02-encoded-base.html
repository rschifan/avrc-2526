11f — Stage 2: Encoded base

Three encoding scales on top of stage 1: node radius from bc, link stroke width and opacity from weight, node fill from community. Suddenly the picture has information density. Source tutorial →

What this stage adds (vs the previous stage)

Three scales, three encodings — the visible delta is large but the code delta is four lines and an updated .attr on each mark.

  • Node radius: scaleLinear().domain(extent(bc)).range([3, 14]). Linear is used here for simplicity; 11d argues for scaleSqrt (so visual area, not radius, grows with the metric). In practice either works for this dataset because the bc range is narrow.
  • Link stroke width: scaleLinear().domain(extent(weight)).range([0.6, 4]). Stronger co-occurrences become more visible lines.
  • Link opacity: scaleLinear().domain(extent(weight)).range([0.3, 1]). Pairs with stroke width — light, narrow edges for occasional co-occurrences; dark, wide ones for frequent pairs. Together they reduce visual noise from the long tail of weak edges.
  • Node fill from community: scaleOrdinal(schemeTableau10). The same pattern as 11d §1, applied to stage 1's previously uniform palette.
  • Why the order matters: encoding decisions are independent concerns. You could add them in any order, but isolating them one at a time (radius first, colour first, opacity first) is a good debugging strategy when building from scratch — each one is verifiable on its own.
