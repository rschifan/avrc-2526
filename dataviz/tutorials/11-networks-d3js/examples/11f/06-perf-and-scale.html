11f — Stage 6: Perf and scale instrumentation

An FPS counter and three throttling knobs — the first optimizations to reach for before switching renderer. With 187 nodes it's overkill; the point is to see the knobs so you can apply them at 2000. Source tutorial →

What this stage adds (vs the previous stage)

Three SVG-side performance levers, in order of impact: a tick budget (don't repaint every tick), idle freeze (stop work once cool), and label-on-demand (only update labels when at least one is visible). Each buys you ~2× the workload before SVG starts to choke.

  • Tick budget: if (++tickCount % N) return; skips paint frames. N = 2 halves the per-tick DOM work; the simulation still updates positions every tick (visually smoother than throttling the simulation itself). Drag responsiveness starts to suffer above ~3.
  • Idle freeze: once alpha < alphaMin the simulation fires "end" and stops emitting ticks — so the FPS counter naturally drops to zero. The overlay reads "idle (frozen)".
  • Label-on-demand: when every label is hidden (e.g. zoomed out), skip the label.attr("x", …) update. With 187 labels it saves a few hundred attribute writes per tick.
  • vector-effect: non-scaling-stroke: lets you scale the wrapper <g> without recomputing per-element stroke widths. Used by every L11 example; mention here because in a pure-perf context it's a free 5–10% speedup on dense edge sets.
  • What this is not: a quadtree-aware spatial index, a different renderer (Canvas / WebGL), or a Web Worker offload — those are bigger changes. 11h shows the Worker option; the numbers below show where the simple SVG pipeline starts to struggle.
  • How to read the FPS counter: 60 = monitor refresh ceiling. On this dataset the simulation runs at 60 for several seconds, then cools and emits the "end" event; the overlay switches to "idle (frozen)". If you were on 2000 nodes you'd see ticks barely keeping up with the simulation — that's the canvas signal.
