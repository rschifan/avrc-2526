11e §2 — Styled label boxes (foreignObject)

SVG <text> is plain text. For pill-shaped, coloured, shadowed labels we need HTML/CSS — embedded into the SVG via <foreignObject> wrapping an xhtml:div. Source tutorial →

What this example shows

Replaces SVG <text> with a richer construct that lets each label carry its own CSS — background colour, border-radius, padding, drop shadow, real font stack. Only the 12 most-central characters get labels — the curated set worth the extra layout cost.

  • What is foreignObject: a special SVG element that lets you embed foreign markup (XHTML, MathML) inside SVG. The child lives in HTML's box-model world; everything outside lives in SVG-land. Position with x, y, width, height attributes; style the inside with regular CSS.
  • The xhtml: namespace prefix: when D3.js's append("xhtml:div") runs, the resulting element is in the XHTML namespace — required so the browser parses it as HTML and applies CSS, instead of as SVG (where most CSS properties are no-ops).
  • Why not just SVG <text>: SVG text doesn't wrap, doesn't have padding, doesn't have border-radius, and styling is finicky. For one-line labels with a coloured pill background, foreignObject is the path of least resistance. For thousands of labels on a dense plot, plain <text> is much faster — pick based on count.
  • Positioning: fo is positioned with its top-left at the given (x, y); to centre the label above the node we set x = node.x - W / 2 and y = node.y - H - r - 2. The pointer-events: none ensures the label doesn't intercept clicks meant for the node.
  • Drop shadow via inline CSS: box-shadow works on HTML divs but not on SVG elements. This is the practical reason to switch — for visual polish that SVG can't easily provide.
  • Performance caveat: every foreignObject spawns a small subtree of HTML the browser must lay out on every position update. Fine for 12 labels; sluggish for 187. Save it for the curated set.
