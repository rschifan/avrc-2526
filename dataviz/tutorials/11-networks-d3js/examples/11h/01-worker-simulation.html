11h — Web Worker simulation

Same simulation, two threads, same SVG renderer in both panes. The top pane runs d3.forceSimulation on the main thread — alongside the rendering. The bottom pane runs the simulation in a Web Worker; the main thread only paints the positions it receives. Source tutorial →

What this example shows

Both panes render to SVG (one <line> per edge, one <circle> per node). The renderer is identical; the only difference is where d3.forceSimulation runs. The worker file at ../../_shared/force-worker.js imports d3 from the CDN and communicates with this page via postMessage.

  • Transferable typed arrays: positions are shipped as a Float32Array.buffer listed in the second argument of postMessage — ownership is transferred (O(1)) instead of cloned (O(n)). At 2000 nodes that's the difference between a 16 KB copy 60 times a second and a pointer handoff.
  • Endpoint table shipped once: link source/target indices arrive in a single "links" message at startup. Per-tick payloads carry only positions.
  • SVG marks bound to indices: each pane creates one <line> per edge and one <circle> per node up front, with data bound to integer indices. The tick handler reads positions out of the typed arrays and writes x1/y1/x2/y2 and cx/cy attributes — same pattern every other L11 example uses.
  • The drag protocol exists but isn't wired here: the worker accepts drag-start / drag-move / drag-end messages (see _shared/force-worker.js); the main thread would post them with the dragged node's id and current pointer coordinates. This demo focuses on the simulation-on-thread question alone.
  • How to read the stats line: fps is bounded at 60 regardless of how heavy a tick is, so the honest signal is the ms/tick on the left and ms/render on the right. The left value covers both simulation and rendering; the right one covers rendering only (the simulation cost is spent on a different thread). At small n they're similar; as n grows, the left climbs faster.
  • Worker prerequisite: this file must be served over HTTP, not opened from file://. The browser blocks new Worker() in file:// contexts. Use python -m http.server.
