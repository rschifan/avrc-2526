10e §3 — Naïve zoom: transform a wrapping <g>

Wrap the dots in a group and write the zoom transform onto it. Wheel zooms, drag pans, double-click zooms in — but the axes live outside the wrapper, so as you zoom in the dots drift away from the tick labels. This is the wrong pattern for any chart with axes; the next example fixes it. Source tutorial →

Try wheel-zooming or dragging. Notice the axis numbers stay put while the dots scale away from them — the labels no longer describe the data underneath.

What this example shows

The simplest way to wire up d3.zoom: attach the behaviour to the SVG, and on every zoom event write the resulting transform onto a wrapper group that contains the marks. SVG composes the parent's transform attribute with its descendants, so the whole subtree scales and translates as one rigid object.

  • d3.zoom() is a behaviour object. You configure it (scaleExtent, extent, translateExtent) and then attach it with selection.call(zoom). The behaviour silently installs wheel, drag, double-click and pinch listeners on that element.
  • The "zoom" event's transform is an object with three numbers: k (scale factor, ≥ 1 here), x and y (pixel translations). Its toString() produces a valid SVG transform string like translate(120, 80) scale(2.5), so assigning it to the transform attribute applies it directly.
  • Why this is "naïve": the axes were appended to plot, not inside the wrapper group, so they never receive the transform. Dot positions move; tick labels stay still. After any zoom the tick labels stop describing the data underneath — fatal for any chart with a quantitative axis.
  • Side-effect on radius: because the whole group is scaled by k, the dots' r attribute is visually multiplied too. Zoom in 16× and a 5-pixel dot becomes 80 pixels wide. Sometimes desirable (treemaps, networks); usually not what you want for scatter.
  • Pattern still useful for diagrams without axes — force-directed graphs, treemaps, chord diagrams. That is exactly how L11 and L12 use it.
