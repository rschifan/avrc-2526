12f — Beyond D3: the network visualization ecosystem

L11 and L12 have shown how far you can take a network in pure D3: force layouts up to ~2k nodes in SVG, with perf instrumentation (11e) and a Web Worker (11f) to push further, and a Canvas renderer for ~10k–20k, plus alternative layouts (matrix, chord, sankey, bundling) when the node-link view fails. Past that point, you reach for specialised tools. This tutorial maps the ecosystem — which library to pick when D3 stops being the right answer.

1. The decision tree

The three questions that drive the choice:

  1. How big is the graph? Order of magnitude — 100, 1k, 10k, 100k, 1M+ nodes. Each ceiling pushes you up a tier.
  2. Do you need network analysis or just rendering? Pure rendering tools draw a graph fast; "graph platforms" also run centrality, community detection, shortest paths in the same library.
  3. How custom does the visual encoding need to be? D3 is the most flexible (paint anything onto SVG / Canvas); higher-level libraries make the common case easy at the cost of customisation.

2. The libraries

Library Renderer Realistic ceiling Built-in analysis Sweet spot
d3-force (this course) SVG / Canvas ~10k nodes (Canvas + 11f Worker) None — pair with networkx server-side Custom-encoded network visualisations, full visual control.
Cytoscape.js Canvas (default) / WebGL ~50k nodes interactive Graph theory algorithms, layouts, selectors Domain apps that need a "graph database UI": bioinformatics, infrastructure maps, BPMN. Stylesheet-driven (CSS-like syntax for graph elements).
sigma.js + Graphology WebGL ~100k nodes interactive Algorithms via Graphology (separate library) Large-scale network browsers — Gephi's web cousin. Sharp separation between "the graph data structure" (Graphology) and "the renderer" (sigma).
Cosmograph WebGL (GPU force layout) 1M+ nodes None — pure renderer "I have a million nodes, I need to look at them now." Force layout runs on the GPU; everything stays interactive.
Gephi (desktop) OpenGL desktop app ~500k nodes Full suite — modularity, ForceAtlas2, statistics Exploratory analysis in a notebook-style desktop GUI. Export → publication or screenshot, not embed-in-page.
vis-network Canvas ~5k nodes Minimal Quick demos and dashboards — declarative options-object API, less flexible than D3 but ~10 lines to working chart.
PyVis HTML wrapper over vis-network ~5k nodes NetworkX integration "I have a NetworkX graph in a Jupyter notebook and I want to look at it interactively right now."

3. Concrete recommendations

  • You are in this course's regime (≤ 10k nodes, custom encoding, embedded in a web page): D3, with the L11 patterns.
  • You inherited a NetworkX graph and want to inspect it: PyVis is three lines (nt = Network(); nt.from_nx(G); nt.show("g.html")). Acceptable for ≤ 1k nodes; quality drops fast above that.
  • You're building a domain product with a graph as the central object (citation manager, knowledge base, BPMN editor): Cytoscape.js. The stylesheet API and the built-in algorithms save weeks of work.
  • You're shipping an explorer for 50–500k nodes: sigma.js + Graphology. The Gephi-style "load a big network, navigate it" pattern.
  • You have a million-node graph and need it real-time: Cosmograph. The only realistic option short of writing your own WebGL renderer.
  • You're doing one-off exploratory analysis, no web embed needed: Gephi. Still the best desktop tool for "let me poke at this graph".

4. The Graphology / sigma split

Worth its own section. Graphology is a graph data structure with a rich algorithm library (centrality, community, traversal, generators) — it does not render. Sigma.js renders Graphology graphs to WebGL. Together they are the closest thing to "NetworkX + matplotlib for the browser":

// graphology
import Graph from "graphology";
import { density, modularity } from "graphology-metrics";
import louvain from "graphology-communities-louvain";

const graph = new Graph();
graph.addNode("alice", { x: 0, y: 0, size: 5 });
graph.addNode("bob",   { x: 1, y: 1, size: 5 });
graph.addEdge("alice", "bob");

louvain.assign(graph);                          // adds .community attribute
console.log(density(graph), modularity(graph)); // numbers
// sigma
import Sigma from "sigma";
const renderer = new Sigma(graph, document.getElementById("container"));
// that's the entire renderer. Tens of thousands of nodes, interactive.

The architecture lesson: separate the data structure from the renderer. D3's data join blurs the two; that's fine for flexibility, painful when the renderer is the bottleneck. Graphology/sigma is the orthogonal design — data and rendering each evolve independently.

5. When to stay in D3

The pitch above is mostly about leaving D3. The counter-argument: D3 is the most flexible option for visualisations that are not off-the-shelf node-link diagrams. A few categories where D3 stays the right answer at any scale:

  • Multi-view dashboards where the network is one panel among many (timeline, map, scatter, table). D3 composes naturally; embedding sigma.js into a larger D3 layout is fighting the API.
  • Bespoke visual encoding — chord diagrams, sankey, edge bundling, arc diagrams. All of L12. The other libraries focus on node-link.
  • Print-quality SVG export. D3 produces SVG you can ship to a publication. WebGL renderers need a screenshot path.
  • Animation-heavy storytelling — the transition vocabulary from 10b and the patterns in 12e (snapshots, streaming). D3 transitions interpolate cleanly across these states; sigma's animation API is much narrower.
Takeaway. D3 is the right answer for custom-encoded networks up to ~10k nodes with a Canvas renderer and the 11e/11f scale story; for larger graphs or off-the-shelf node-link UIs, Cytoscape.js (≤ 50k), sigma.js+Graphology (≤ 500k), or Cosmograph (≥ 1M) each have their place. The architectural insight worth carrying: separate the data structure from the renderer. That's the design that lets sigma render what Graphology stores, and it's the same design that lets you push d3-force into a Web Worker and Canvas into a separate paint path.
