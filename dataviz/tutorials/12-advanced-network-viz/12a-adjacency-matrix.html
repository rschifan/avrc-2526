12a — Adjacency matrix

A node-link diagram is the obvious way to draw a network — but as the graph grows, the edges turn into a hairball. The adjacency matrix is the second standard view: every node gets a row and a column, and each cell encodes whether (and how strongly) the two are connected. It scales much better, and the right ordering makes communities pop out as visible blocks.

1. From edge list to matrix

The L11 JSON has nodes and an edge list. Building an N × N matrix is one nested loop:

const N = nodes.length;
const matrix = Array.from({ length: N }, () => new Array(N).fill(0));
const idx = new Map(nodes.map((d, i) => [d.id, i]));

links.forEach(l => {
    const i = idx.get(l.source);
    const j = idx.get(l.target);
    matrix[i][j] = l.weight;
    matrix[j][i] = l.weight;        // undirected → symmetric
});

Once we have the matrix, drawing it is a flat data join. We flatten it into one rectangle per non-zero cell and bind:

const cells = matrix.flatMap((row, i) =>
    row.map((v, j) => ({ i, j, v })).filter(c => c.v > 0));

cellG.selectAll("rect").data(cells).join("rect")
    .attr("x", c => x(c.j))           // x = column band scale
    .attr("y", c => y(c.i))           // y = row band scale
    .attr("width",  x.bandwidth())
    .attr("height", y.bandwidth())
    .attr("fill", c => colour(c.v));

2. Why ordering is everything

The matrix below shows the top-40 GoT characters by centrality. Switch the row/column order with the radio buttons:

  • Alphabetical — the default; cells are scattered, no visible structure.
  • By community — characters from the same Louvain community sit next to each other; communities show up as dark blocks along the diagonal.
  • By centrality — most central first; the top-left corner concentrates the heaviest edges.
  • Barycenter — the classic block-revealing reordering: each row is moved next to the average position of its neighbours. Iterate a few passes and the matrix's natural blocks emerge from the data alone, no community labels needed.

The matrix is the same; only the index permutation changes. That's the whole point — for matrix views, ordering is the visualization. (Mike Bostock's Les Misérables co-occurrence matrix is the canonical demo of this.)

Reading a cell. Each filled square is one edge, and its colour runs light → dark with the edge weight (how many chapters the two characters share). Community is not drawn on the cells — that just clutters them — but on the thin coloured strips along the top and left edges, one per character in its community's colour. Order by community and those strips line up into matching-colour runs that frame the diagonal blocks. The legend just above the matrix maps both encodings.

3. Row + column highlight on hover

Reading a single relationship in a matrix is harder than in a node-link view: you have to find the row, find the column, and trace your eye to the intersection. A hover handler that lights up the active row and column makes the cross-reference effortless:

cells.on("mouseenter", (e, c) => {
    rowMarker.attr("y", y(c.i)).attr("opacity", 1);
    colMarker.attr("x", x(c.j)).attr("opacity", 1);
});
cells.on("mouseleave", () => {
    rowMarker.attr("opacity", 0);
    colMarker.attr("opacity", 0);
});

The matrix above already wires this up — hover any cell and the two grey strips show which row and column are involved; the corner labels switch to bold for the matching characters.

Takeaway. A matrix view is just a 2D grid of cells with a colour scale. The single decision that makes or breaks it is row ordering: alphabetical is useless; community-based ordering surfaces the block structure. Use the matrix when the node-link view is too dense, when edge weights matter, or when the question is "who connects to whom" rather than "what's the global shape".
