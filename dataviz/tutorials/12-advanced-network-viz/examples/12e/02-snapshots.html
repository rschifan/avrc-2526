12e §5 — Snapshot transitions

A small research group sampled once a year. The three snapshots share most members; surviving nodes keep their positions across each swap, newcomers grow in, departed members fade out. Source tutorial →

What this example shows

The snapshot pattern: discrete data states, not a continuous stream. Each snapshot ships its own nodes and links; we map old x / y onto surviving ids via a byId Map so the simulation continues from the previous layout instead of restarting.

  • The byId trick: at every snapshot we copy the previous frame's x / y onto the new node objects (by id). The simulation reads these as starting positions and treats the layout as "mostly relaxed" — a small alpha (0.4–0.5) is enough.
  • Keyed joins for both nodes and links: nodes by d.id; links by "source-target". Without this every snapshot would re-create every DOM element from scratch.
  • The update callback re-encodes the fill, because group membership can shift between snapshots — watch member F change community in Year 3.
  • When to use this vs streaming: snapshots when you have a known, small number of states (years, sessions, sliding windows); streaming when events arrive at unpredictable times.
  • Real-world data: temporal co-authorship networks (DBLP), evolving citation graphs, friendship snapshots over months — same pattern, only the source of the arrays changes.
