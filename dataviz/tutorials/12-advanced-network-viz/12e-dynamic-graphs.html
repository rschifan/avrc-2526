12e — Dynamic and streaming graphs

Every L11 network was static — load the JSON, run the simulation, never touch the data again. But the interesting graphs are rarely still: friendships form, papers cite each other, transactions fire, servers join and leave. A dynamic graph visualization animates that change, and the animation itself carries information a static snapshot cannot — the order in which structure appeared, and how it grew. This tutorial builds two production patterns on top of the L11 force layout: streaming (the graph grows one event at a time) and snapshots (the whole graph swaps between known states).

1. Why animate a changing graph?

A still picture of a network answers "what is connected to what". Showing the network evolve answers questions a single frame cannot:

  • Social & collaboration networks. Follows, friendships, and co-authorships accumulate over time. Animation shows when communities crystallise and which early members become the bridges between them.
  • Citation & knowledge graphs. As papers and entities link up, a few seminal nodes attract far more connections than the rest — you can literally watch the hubs emerge (the demo in §4 reproduces exactly this effect).
  • Live monitoring. Network traffic, payments, or message flows arrive as a stream of edges. A dynamic view surfaces bursts and anomalies as they happen, not in a next-day report.
  • Infrastructure & systems. Service meshes and device topologies gain and lose nodes constantly; animating the churn makes a cascading failure visible as it spreads.

The throughline: motion encodes sequence, emergence, and churn. The engineering problem is to make that motion legible — surviving nodes must stay put so the eye can track them, while new and departing nodes animate in and out. That is almost entirely a data-join problem, which is why the L10/L11 join is the foundation here.

2. Three flavours of "dynamic"

"Dynamic graph" is an umbrella; the right code depends on which sub-pattern you have:

PatternSourceCode shape
Streaming An event stream — a new node arrives, an edge appears, a node leaves. Append to the nodes / links arrays; call sim.nodes(nodes), rebuild forceLink(links), restart at a moderate alpha.
Snapshots Discrete time slices — mostly the same nodes, different edge sets, like yearly co-authorship networks. Swap the entire arrays; rely on the key function in selection.data(…, key) so D3 matches surviving nodes to their old positions instead of restarting from zero.
Continuous One running simulation, attribute updates only (no add/remove). Mutate per-datum properties; let the tick handler pick up the new values. The force simulation is untouched.

The first two need new mechanics; the third is just attribute updates and works out of the box.

3. The core mechanic: keyed joins

Static data(arr).join(tag) matches by index by default — element 0 binds to arr[0], element 1 to arr[1], and so on. Add a node to the front of the array and every existing circle is rebound to the wrong datum: positions scramble, colours jump. The fix is a key function that ties each DOM element to a stable identity:

const node = nodeG.selectAll("circle")
    .data(nodes, d => d.id)            // ← key by id, not array position
    .join(
        enter  => enter.append("circle")
            .attr("r", 0)                          // new arrivals grow in…
            .call(e => e.transition().duration(400).attr("r", d => r(d.degree))),
        update => update                           // survivors: leave position to the sim
            .call(u => u.transition().duration(400).attr("r", d => r(d.degree))),
        exit   => exit.transition().duration(400).attr("r", 0).remove()   // departures shrink out
    );

The three-callback form of selection.join (from 10a) maps one-to-one onto the three things a changing graph does: enter nodes grow from r = 0; update nodes keep their identity (the simulation owns their position, but we can still re-encode size or colour as their data changes); exit nodes shrink and remove themselves. Keep the simulation state on the surviving datum objects and the layout never restarts from scratch — that stability is what makes the animation readable.

4. Streaming — watch a network grow

The most compelling thing a dynamic graph can show is structure emerging from a simple rule. Our stream follows preferential attachment (the Barabási–Albert model): every arriving node links to a few existing ones, choosing them with probability proportional to how many connections they already have. "The rich get richer" — and a handful of hubs pull away from the crowd. Watch the node size (it tracks degree) and the largest hub counter as the graph grows.

Step 1 — the model. The classic trick for a degree-weighted random pick is an endpoints pool: every node id appears in it once per incident edge, so a uniform draw from the pool is automatically proportional to degree.

const M = 2;                 // edges each new node brings in
const endpoints = [];        // node ids, repeated once per incident edge

function addNode() {
    const v = { id: nextId++, degree: 0 };
    const targets = pickTargets(M);          // ∝ degree (preferential attachment)
    nodes.push(v);
    for (const t of targets) {
        links.push({ source: v.id, target: t });
        endpoints.push(v.id, t);             // grow the weighted pool
        v.degree++; byId.get(t).degree++;
    }
    update(v.id);                            // redraw + reheat, flag the new arrival
}

function pickTargets(m) {
    if (nodes.length <= m) return nodes.map(n => n.id);   // bootstrap the seed
    const chosen = new Set();
    while (chosen.size < m)
        chosen.add(endpoints[Math.floor(Math.random() * endpoints.length)]);
    return [...chosen];
}

Step 2 — one update(). Every change runs the same four steps: hand the new array to the simulation, rebuild the link force, reheat gently, and re-run the keyed joins. The radius scale is re-domained on each call so the size encoding keeps pace with the growing hubs.

function update(newestId) {
    r.domain([1, d3.max(nodes, d => d.degree) || 1]);     // degree → radius
    sim.nodes(nodes);                                      // tell the sim about new nodes
    sim.force("link", d3.forceLink(links).id(d => d.id).distance(30));  // rebuild, don't mutate
    sim.alpha(0.5).restart();                              // gentle reheat, not 1.0

    node = nodeG.selectAll("circle").data(nodes, d => d.id).join(/* enter / update / exit */);
    link = linkG.selectAll("line")
        .data(links, l => `${l.source.id ?? l.source}-${l.target.id ?? l.target}`)
        .join(/* … */);
    pulse(newestId);                                       // flash the arrival
}

Three details worth internalising:

  • Rebuild the link force, don't mutate it. forceLink captures the array at construction; appending to it afterwards is a no-op until you attach a fresh forceLink(links). It is cheap — do it on every update.
  • Reheat with a small alpha (≈ 0.5), never 1.0. The layout is mostly settled; you only need a nudge to absorb the newcomer. Full reheats look explosive.
  • Key the link join too. Lines are keyed by a derived source-target string, so existing edges are left alone and only genuine additions animate in.

5. Snapshots — replay a timeline

The second pattern handles discrete states, each with its own node and edge set that share most members — a research group sampled once a year, say. We want each transition to preserve the positions of nodes present in both frames, fade in newcomers, and fade out anyone who left, so the eye can follow who stayed and who moved.

The mechanism is the same key-by-id join — D3 matches surviving nodes by id, the simulation keeps their x / y, and only enter/exit nodes get fresh positions:

function snapshot(state) {
    // carry previous x/y onto surviving ids so the layout continues, not restarts
    nodes = state.nodes.map(d => Object.assign(byId.get(d.id) ?? {}, d));
    links = state.links.map(l => ({ ...l }));
    byId = new Map(nodes.map(d => [d.id, d]));

    node = nodeG.selectAll("circle").data(nodes, d => d.id).join(/* enter / update / exit */);
    link = linkG.selectAll("line").data(links, l => `${l.source}-${l.target}`).join(/* … */);

    sim.nodes(nodes);
    sim.force("link", d3.forceLink(links).id(d => d.id).distance(40));
    sim.alpha(0.4).restart();
}

The byId map is the crux: it carries each surviving node's previous x / y across the swap. Without it the simulation would re-seed every node at the centre on every frame and the whole layout would jump; with it you get smooth move-in / move-out and a stable mental map across the timeline.

6. Performance and reheating

Dynamic graphs are harder than static ones because every change reheats the simulation, and reheats are the most expensive moments of a force layout. Three knobs matter more here than for static graphs:

  • Reheat with a small alpha. sim.alpha(0.3).restart() instead of sim.alpha(1) when the graph is mostly stable. Use alphaTarget(0.05) for an "always-warm" mode that keeps a live stream gently flexible without ever fully cooling.
  • Cool to zero when input stops. The drag pattern from 11c uses alphaTarget(0).restart() on end; do the same after the last event so the CPU goes idle instead of spinning on an already-settled layout.
  • Batch high-rate streams. If events arrive faster than the simulation can absorb them, collect for ~100 ms and call update() once per batch. The eye gets a readable rhythm and the simulation gets stable ticks between rebuilds.

7. Beyond force-directed

The snapshot pattern composes with any L12 layout, not just force-directed:

  • Adjacency matrix (12a) — the cleanest case. Cells are positions and rows/cols are ordered by some attribute; when the graph updates you re-bind cells with a join keyed by (source, target). No simulation to reheat — the data join plus a row/col reorder transition is enough.
  • Chord diagram (12b) — ribbons interpolate smoothly between weight matrices via a d3.transition on the d3.ribbon path.
  • Sankey (12c) — positions are laid out per snapshot; transitions animate x0 / x1 / y0 / y1 on each rect.

The common thread: matrix- or layout-shaped data is easier to animate honestly than a force-directed view, because positions are deterministic functions of the data. Force layouts are stable across small changes but jitter under large ones — which is exactly why the keyed join and the gentle reheat matter so much.

Takeaway. A dynamic graph is a static graph plus a keyed join and a thoughtful reheat. Streaming appends one element per event and nudges the layout; snapshots swap the whole data set but preserve positions for survivors. Encode the thing that is changing (here, degree → size) so the process becomes visible — that is what turns a moving picture into an explanation.
