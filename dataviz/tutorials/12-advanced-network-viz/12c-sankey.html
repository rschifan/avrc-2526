12c — Sankey diagram

Node-link, matrix, chord all assume the network is a relation — A is connected to B. A Sankey assumes the network is a flow — A passes some quantity to B, which passes part of it on to C. Energy budgets, web-traffic funnels, migration paths, allegiance shifts: any DAG with weighted edges fits the shape.

1. The data shape

Sankey input is a node-link object — same as 11a — but with a constraint: the graph must be a directed acyclic graph. Each node belongs to a layer; edges only go forward.

const data = {
    nodes: [
        { id: "Stark"    },  { id: "Lannister" },  { id: "Targaryen" },
        { id: "Loyalist" },  { id: "Defector"  },  { id: "Outsider"  },
        { id: "Survives" },  { id: "Dies"      }
    ],
    links: [
        { source: "Stark",    target: "Loyalist", value: 14 },
        { source: "Loyalist", target: "Survives", value: 16 },
        ...
    ]
};

The GoT co-occurrence graph is undirected and full of cycles, so it isn't Sankey-shaped. The demo below uses a small synthetic 3-stage flow inspired by character allegiance shifts.

How to read a Sankey. Read it left to right. Our toy flow has three stages: a character's origin house (Stark, Lannister, Targaryen) → the allegiance they take (Loyalist, Defector, Outsider) → their eventual fate (Survives, Dies). Every node is a rectangle whose height is its total throughput, and every band is one source→target flow whose width is proportional to the quantity — here, the number of characters following that path. Because flow is conserved, the bands entering an internal node sum to exactly the bands leaving it: a tall "Loyalist" rectangle means many characters converged on that allegiance, and you can trace how that pool then splits between Survives and Dies. Width = quantity is the whole grammar — you compare magnitudes by eye and follow where a quantity merges and splits.

2. The d3-sankey plugin

D3 core ships layouts for trees, packs, treemaps — but not Sankey. The d3-sankey plugin provides the layout. It mutates each node (x0, x1, y0, y1) and each link (width proportional to value):

const sankey = d3.sankey()
    .nodeId(d => d.id)
    .nodeWidth(15).nodePadding(12)
    .nodeAlign(d3.sankeyJustify)
    .extent([[0, 0], [innerW, innerH]]);

const { nodes, links } = sankey(data);

Then rendering is two ordinary joins (links as paths via d3.sankeyLinkHorizontal(), nodes as rects):

g.append("g").attr("fill", "none").attr("stroke-opacity", 0.4)
    .selectAll("path").data(links).join("path")
        .attr("d", d3.sankeyLinkHorizontal())
        .attr("stroke-width", d => Math.max(1, d.width))
        .attr("stroke", d => colour(d.source.id));

g.append("g").selectAll("rect").data(nodes).join("rect")
    .attr("x",      d => d.x0)
    .attr("y",      d => d.y0)
    .attr("width",  d => d.x1 - d.x0)
    .attr("height", d => d.y1 - d.y0)
    .attr("fill",   d => colour(d.id));

3. Tuning the layout

Pick a node-alignment strategy (the radio buttons below) and watch how nodes snap to layers:

  • sankeyLeft — every node hugged to its earliest possible layer (left).
  • sankeyRight — every node hugged to its latest possible layer (right).
  • sankeyJustify — sources left, sinks right, internals in between.
  • sankeyCenter — internal nodes float toward the middle.

4. Synchronized highlighting across two views

The same data can be drawn two ways at once. A production pattern: a Sankey on top, a force-directed view of the same nodes underneath, and a single hover handler that lights up the focal node and its incident edges in both panels simultaneously. Hover any rectangle in either chart:

↗ example

Notice what each view gives up. The Sankey encodes two things the force view cannot: left-to-right stage order and width = quantity. The force layout in the lower panel shows the identical eight nodes and thirteen flows, but it has no notion of stages or magnitude — it only balances springs, so the clean three-stage structure collapses into a ball. That contrast is the whole reason Sankey exists for flow data; pairing the two with linked hover lets you read magnitudes off the Sankey and free-form adjacency off the force view at once.

The trick is to keep one set of hover dispatchers and update both chart layers from a single function:

function highlightAcross(node) {
    const id = node?.id;
    sankeyNodes.classed("focus", d => d.id === id)
               .classed("faded", d => id && d.id !== id && !connected(d, id));
    sankeyLinks.classed("focus", l => id && (l.source.id === id || l.target.id === id))
               .classed("faded", l => id && l.source.id !== id && l.target.id !== id);
    forceNodes.classed("focus", d => d.id === id) /* …same idea… */;
    forceLinks.classed("focus", l => ...) ;
}

5. Tooltips with in / out degree

A small floating tooltip on hover gives a per-node summary. Pre-computed degrees keep it fast:

const inDeg  = d3.rollup(data.links, v => d3.sum(v, d => d.value), d => d.target);
const outDeg = d3.rollup(data.links, v => d3.sum(v, d => d.value), d => d.source);

el.on("mouseenter", (e, d) => tip
    .style("opacity", 1)
    .html(`<b>${d.id}</b><br>in: ${inDeg.get(d.id) ?? 0} · out: ${outDeg.get(d.id) ?? 0}`));
el.on("mousemove", e => tip.style("left", e.pageX + 12 + "px")
                            .style("top",  e.pageY + 12 + "px"));
el.on("mouseleave", () => tip.style("opacity", 0));
Takeaway. Sankey is the right view when the data is a directed, weighted flow. The d3-sankey plugin computes the geometry; rendering reduces to two joins. The pattern that scales is synchronised hover: pair a Sankey with a force or matrix view on the same dataset, and a single dispatcher updates both. That's how production dashboards combine techniques.
